# The Gate Before the Gate

*How running architecture review before creating new AI skills saved months of compounding complexity*

**TL;DR**
- Running /architecture-review before creating a new skill caught 4 failure modes that felt invisible in the moment
- "Extend before create" is now a hard rule: new workflows become flags on existing skills first, only promoted after 3 cross-domain uses
- The real cost of skill bloat isn't discoverability -- it's every future session paying the context tax for something you built once and never reused

---

Three days ago I almost built a 41st skill.

The session was going well. I'd just used Jarvis to execute a full car-buying research pipeline -- market intel, tariff analysis, dealer ranking, four personalized outreach emails, distributed to Slack for mobile copy-paste. It worked cleanly. My instinct was immediate: formalize this as `/vendor-outreach`.

I almost just did it. That's the ADHD build velocity pattern I've logged before -- when something works, the energy is high and the obvious next move feels like the right one. Instead I ran `/architecture-review` first.

Three parallel agents. First-principles, fallacy detection, red-team. All three converged on the same answer: don't create a standalone skill.

The arch review surfaced things I couldn't see while riding the high of a successful session. Recurrence inflation -- I'd rated the workflow "High recurrence" when it was actually Medium (buying a car happens once every few years, not weekly). A premature dependency on Gmail MCP that didn't exist yet. Prompt injection risk in the research-to-email pipeline I hadn't thought through. And negotiation data leakage -- personalized dealer emails contain pricing intelligence that shouldn't flow through the same pipeline as general research.

None of these are fatal problems. All of them are significantly cheaper to not-build than to build-and-fix.

The resolution was cleaner than the rejected alternative: `/research --outreach` mode. The workflow I wanted lives as a flag on a skill Eric already knows. It's discoverable because it extends something familiar. It has a promotion trigger built in -- if the pattern gets used 3 times across different contexts (cars, job offers, vendor contracts), it earns a standalone skill. Until then, it stays embedded.

This produced a second principle worth naming explicitly: **extend before create**. When a new workflow emerges, the first question isn't "should this be a skill?" It's "can this be expressed as a mode on something that already exists?" The answer is yes more often than it feels like in the moment.

The broader lesson is about gates. I already had a steering rule that said: before any hard-to-reverse decision, run `/architecture-review`. Skill creation wasn't listed explicitly as a trigger -- it felt lighter than architecture or tool adoption. It isn't. A skill you create becomes a permanent fixture in every future session's context window. It shapes how you frame problems. It accretes. The cost isn't creation, it's the tax every subsequent session pays whether the skill is relevant or not.

The gate I'm adding: `/architecture-review` is now a required step before `/create-pattern`. Not recommended. Required. The few minutes of review cost is nothing against the compounding cost of a skill that didn't need to exist.

Forty skills is the current count. Each one survived a tighter filter than the last. That's the right direction.

What you can apply today: before formalizing any repeatable workflow as a named tool or process, run it through the "extend before create" check. Ask what already exists that could absorb this. Set a usage threshold before promotion. Build the gate before you build the thing.
