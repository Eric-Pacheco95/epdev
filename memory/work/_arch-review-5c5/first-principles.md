# First-Principles Analysis: 5C-5 Overnight Runner as Backlog Producer

**Date:** 2026-04-02
**Reviewer:** Jarvis (Sonnet 4.6)
**Proposal:** Make overnight runner a producer that emits `backlog_append()` tasks after completing its iterative loop.

---

## 1. What Is the Fundamental Problem?

### What "visibility" actually requires

The stated motivation is making overnight outputs visible to the dispatcher. But this conflates two distinct problems:

**Problem A — Traceability:** Today, when the overnight runner completes a dimension (e.g., `scaffolding` improved by 3 kept iterations), the changes live on a branch (`jarvis/overnight-YYYY-MM-DD`). There is no mechanism to surface "this branch has improvements worth reviewing or applying" into the dispatcher's work queue. The human must notice the Slack summary and manually create follow-up tasks.

**Problem B — Follow-through:** The overnight runner generates iterative micro-improvements (individual commits) plus a quality/security gate result. The improvements are committed but never applied to main. There is no bridge between "overnight produced a good result" and "dispatcher integrates or validates that result."

These are different problems:
- Problem A is a notification/discovery gap — the output exists but no consumer acts on it.
- Problem B is a lifecycle gap — overnight's outputs are orphaned branches with no defined fate.

The proposal (emit backlog tasks) solves Problem A, but only partially addresses Problem B — unless the emitted task explicitly describes what to do with the branch (e.g., review and merge, or re-implement in main).

### The real fundamental question

**Does the overnight runner's output need to be acted on autonomously, or reviewed by Eric?**

- If the overnight runner is doing micro-refactoring (metric-driven iterative changes), those changes are already committed on a branch. The "task" for dispatcher is: merge, reject, or cherry-pick. That is a human review decision, not an autonomous execution.
- If the overnight runner is identifying patterns (e.g., "12 dead functions in codebase_health"), those observations could generate dispatcher tasks that implement the identified fixes. That is actionable autonomously.

The proposal blurs these two cases. Clarity here determines which alternative is correct.

## 2. Irreducible Requirements for Overnight -> Backlog Convergence

For the integration to be correct and safe, the following requirements are non-negotiable:

1. **Atomicity of decision:** A backlog task must represent one bounded unit of work with a single ISC outcome. If overnight produces multi-concern output, splitting is required before entering the backlog.

2. **ISC binary-testability at submission time:** The dispatcher's `select_next_task()` will validate ISC before claiming a task. If ISC is ambiguous or depends on dimension-specific context that isn't passed forward, the task will either be skipped or fail ISC verification at the end of execution. ISC must be evaluable without access to overnight's run context.

3. **Source-tagged for auditability:** The `source` field must be `"overnight"` with a reference to the run date and dimension. The current `backlog_append()` writes `source` to the task record — this is already supported.

4. **Idle-is-success preservation:** `backlog_append()` is idempotent only on `routine_id` dedup. If a dimension finds nothing actionable, it must not emit a task. The producer (overnight runner) must implement this gate explicitly — the backlog library will not do it for you.

5. **ISC command allowlist compatibility:** The dispatcher's `classify_verify_method()` and `sanitize_isc_command()` enforce an allowlist of safe verification commands (`grep`, `wc`, `test -f`, `find`, etc.). Any ISC auto-generated by an LLM must produce commands in this allowlist. LLMs frequently emit `python -c` or `ls` for verification — both are blocked. This is a hard failure mode.

6. **No overnight/dispatcher lock contention:** Both systems use `acquire_claude_lock()` from the shared worktree lib. They cannot run concurrently. If overnight emits tasks after its run, dispatcher will be scheduled for the morning window — no contention. But this dependency must be explicit in scheduling, not assumed.

## 3. Assumption Failures in Each Alternative

### Alt 1: One task per dimension

**Wrong assumption: "Each dimension is independent."**
The overnight runner runs all dimensions in a single worktree on a single branch. Dimension 2 (codebase_health) commits happen after dimension 1 (scaffolding) commits — they are entangled on the same branch. A task emitted from dimension 1 that says "merge these scaffolding improvements" cannot be isolated from dimension 2's changes without cherry-picking. The per-dimension task granularity does not map to per-dimension branch granularity.

**Wrong assumption: "Per-dimension ISC quality depends on the overnight worker generating good ISC in real time."**
This is identified as a con but its severity is understated. The overnight runner's LLM-generated ISC will be produced inside a `claude -p` invocation with a constrained prompt. The LLM has no visibility into the dispatcher's `classify_verify_method()` allowlist. ISC will frequently contain `python -c`, `ls`, or free-text "Review:" methods that fail downstream validation. The result is tasks that route to decision rather than backlog, defeating the purpose.

**Wrong assumption: "Up to 6 tasks per run is manageable."**
The dispatcher runs one task at a time with no daily budget cap (in the current system). Six new tasks from overnight plus heartbeat/routines tasks could exhaust the dispatcher for multiple days with overnight-generated work, crowding out human-submitted tasks. Without budget controls, this is a priority inversion risk.

### Alt 2: One summary task per run

**Wrong assumption: "Combined ISC is acceptable."**
CLAUDE.md's ISC Quality Gate rule explicitly prohibits compound criteria ("no compound criteria joined by 'and'"). A summary task would necessarily combine multiple dimension outcomes into one ISC set. The backlog's `validate_task()` does not enforce the compound-criterion rule (it checks for at least one executable verify method, not ISC quality), but the task will be epistemically unsound — the dispatcher worker cannot verify partial success and will either pass trivially or fail noisily.

**Wrong assumption: "One task per night maximum is simpler."**
It is simpler for the producer, but it creates a bottleneck: if overnight runs every night, there will be one pending summary task always in the backlog. The dispatcher must complete this task before the next overnight run produces another. Given the dispatcher's one-task-at-a-time model and tasks that can take 15+ minutes, a growing backlog of unprocessed overnight summaries is a real outcome.

### Alt 3: Conditional per-dimension (hybrid)

**Wrong assumption: "The overnight worker can judge 'actionable vs informational.'"**
This is identified as a con, but the depth of the problem is understated. The overnight worker is an LLM invoked via `claude -p` with a 2-hour hard kill. It has no access to the task gate's routing rules, the autonomy map, or the ISC allowlist. Its judgment of "actionable" will be based on natural language heuristics. The architectural keywords regex in `task_gate.py` (`architect`, `migration`, `redesign`, etc.) exists precisely because LLMs do not reliably self-classify task scope. The overnight worker will produce false negatives (actionable things classified as informational) and false positives (vague suggestions classified as actionable).

**More fundamentally:** Alt 3 adds LLM judgment as a gate inside the dimension run itself. This is the correct intuition, but it should be implemented as a deterministic post-processing step, not as an LLM judgment call. The LLM should emit structured output; a Python script should parse it and decide whether to call `backlog_append()`.

## 4. Right Granularity for Task Emission

### Per-dimension is wrong; per-finding is right

The overnight runner's output has two distinct categories:

**Category 1 — Branch artifacts:** Commits on `jarvis/overnight-YYYY-MM-DD` that represent already-implemented micro-improvements. These are not dispatcher tasks — they are review candidates. The right path is: emit a single `pending_review` task per run (or per quality-gate pass), with a description of the branch and a pointer to the run report. This routes to Eric via the existing `pending_review` pipeline, not to the autonomous dispatcher.

**Category 2 — Identified gaps:** During iteration, the overnight worker may identify patterns it could not fix in the time budget, or issues outside its scope (e.g., a codebase_health pass identifies 8 functions that need tests, but the worker's guard prevents test generation). These are genuine dispatcher candidates — bounded, ISC-testable, autonomous-safe.

The right granularity: emit Category 2 findings as discrete tasks with human-readable descriptions and pre-validated ISC. Do not emit Category 1 as dispatcher tasks.

### Implementation pattern

Instead of having the LLM inside `run_dimension()` decide whether to emit tasks, add a post-processing step:

```
run_dimension() -> returns structured result dict including:
  - kept_count, discarded_count (existing)
  - identified_gaps: list of {"description": str, "isc": [str], "skills": [str]}

After run loop: for each gap in identified_gaps:
  validate gap ISC against allowlist (deterministic Python)
  if valid: backlog_append(gap, source="overnight", status="pending")
  else: log and skip
```

The `identified_gaps` field is populated by parsing the LLM's structured output (`OVERNIGHT_RESULT:` line already exists as a pattern). This keeps the judgment inside the LLM where it belongs, but routes the output through deterministic validation before touching the backlog.

## 5. Should Overnight Generate ISC, or Describe and Let a Separate Step Add ISC?

### Core tension

The overnight runner is an autonomous agent. Asking it to generate ISC that passes a downstream allowlist creates a reliability problem: the agent does not know the allowlist, and ISC errors are silent failures (tasks route to decision or are skipped, not loudly rejected).

### Recommended approach: Structured output + template ISC

The overnight runner should emit structured gap descriptions in a format that maps to a small set of pre-validated ISC templates:

```
gap_type: "add_tests"  -> ISC template: "Test file exists | Verify: test -f {path}"
gap_type: "fix_lint"   -> ISC template: "No lint errors | Verify: flake8 {scope} --count"
gap_type: "remove_dead_code" -> ISC template: "Dead symbol removed | Verify: grep -c {symbol} {path}"
```

The overnight runner emits `gap_type` + parameters. A post-processor fills in the ISC template. This eliminates LLM-generated ISC entirely while preserving the agent's ability to identify what needs fixing.

If a gap does not map to a known template, it routes to `pending_review` (human-authored ISC required), not to the dispatcher.

This is strictly better than having the LLM generate ISC: it is deterministic, allowlist-safe, and auditable.

## 6. Dispatcher Budget Controls: Right Scope for 5C-5?

### Verdict: Independent but sequencing-dependent

Budget controls are not conceptually part of "overnight runner as producer." They are a general dispatcher property needed whenever any high-volume producer (overnight, heartbeat, routines) can flood the backlog.

However, budget controls become urgent if overnight task emission is implemented without them. Six overnight tasks per run + heartbeat tasks + routine tasks could produce 10-15 pending tasks per day, with no throttle on dispatcher execution time.

**Recommendation:** Implement budget controls as a prerequisite gate for 5C-5, not as part of the same ISC. The overnight integration ISC should include "budget controls are in place" as a dependency criterion, not absorb the implementation.

Specifically required:
- `max_tasks_per_source_per_day` (e.g., 3 from "overnight") — prevents run-away production from any single source
- `max_wall_time_per_task_s` (e.g., 900s / 15 min) — prevents individual tasks from consuming the entire scheduling window
- Daily aggregate cap is a secondary nice-to-have; source-level caps are the critical control

These live in dispatcher config, not overnight runner. The overnight runner should remain unaware of dispatcher scheduling constraints.

## Summary: Recommended Architecture for 5C-5

1. **Do not merge** overnight runner and dispatcher. The proposal correctly identifies this as wrong.

2. **Two output channels** from overnight:
   - **Branch artifacts** (Category 1): emit one `pending_review` task pointing to the branch and run report. This is for Eric to review, not the dispatcher.
   - **Identified gaps** (Category 2): emit discrete tasks via `backlog_append()` with template-matched ISC. Validated deterministically before writing.

3. **LLM emits structured gap output;** Python post-processor handles ISC templating and backlog routing. No LLM-generated ISC in the final task record.

4. **Implement Alt 3 (conditional per-dimension)** but with deterministic post-processing, not LLM judgment as the gate.

5. **Budget controls are a prerequisite,** not a parallel feature. Add `max_tasks_per_source_per_day` to dispatcher config before enabling overnight task emission.

6. **ISC template library** (initially small: `add_tests`, `fix_lint`, `remove_dead_code`, `update_docs`) is the key artifact of 5C-5. Without it, overnight ISC will fail dispatcher validation silently and the feature will appear broken.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| LLM-generated ISC fails allowlist validation | High | Silent feature failure | ISC template library (Section 5) |
| Overnight produces tasks faster than dispatcher executes | Medium | Backlog saturation | Source-level daily cap (Section 6) |
| Category 1 branch artifacts become stale | Medium | Dead branches accumulate | `cleanup_old_branches()` already handles 7-day TTL |
| Overnight and dispatcher lock contention | Low | Runtime abort | Scheduling window separation (overnight 4am, dispatcher 9am+) |
| Gap identification produces false positives | Medium | Wasted dispatcher cycles | Alt 3 + deterministic post-processor catches most |

## Decision Required From Eric

Before implementation, one scoping decision:

**Is the goal of 5C-5 to route overnight findings to the dispatcher (autonomous execution), or to route overnight findings to Eric's review queue (human-in-the-loop)?**

- If autonomous: implement Category 2 gap emission with ISC template library + budget controls as prerequisite.
- If review queue: implement Category 1 branch artifact emission as `pending_review` tasks only. Simpler, lower risk, immediately useful.
- If both: Phase the work — review queue first (low risk), autonomous gap execution second (requires template library and budget controls).

The "both" path is likely correct but should be sequenced, not built in one sprint.
