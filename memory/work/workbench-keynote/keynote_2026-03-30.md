### Slide 4: The 7-Phase Algorithm

**Bullets**:
- OBSERVE: gather context and understand current state
- THINK: analyze constraints, identify risks, consider alternatives
- PLAN: define testable acceptance criteria before building
- BUILD > EXECUTE > VERIFY > LEARN: disciplined delivery

**Image**: Horizontal flow diagram showing 7 phases as connected chevrons: OBSERVE (blue) > THINK (purple) > PLAN (orange) > BUILD (green) > EXECUTE (green) > VERIFY (red) > LEARN (gray). A large red X over the arrow that skips from OBSERVE directly to BUILD, labeled "Most AI usage."

**Speaker notes**:
- This is the framework. Seven phases. Every non-trivial task goes through all of them.
- Most AI usage today skips straight from "I have a question" to "build me something."
- That's like writing code without requirements. We'd never accept that in a PR review.
- The algorithm forces you to observe, think, and plan before you build anything.
- And after you build, you verify against the criteria you defined up front.

### Slide 5: The Quality Gate

**Bullets**:
- 6 binary checks before any building begins
- Each criterion: one sentence, testable, state-not-action
- At least one anti-criterion: what must NOT happen
- Every criterion has a verification method attached

**Image**: A gate or checkpoint visual. Left side: vague requirements ("make it good", "handle errors", "be secure") with red X marks. Right side: passing criteria ("Auth tokens expire after 24h | Verify: unit test", "No PII in log output | Verify: grep scan") with green checkmarks.

**Speaker notes**:
- Before any building starts, the acceptance criteria have to pass six checks.
- Is it concise? Is it testable with a clear pass/fail? Does it describe a state, not an action?
- Are there anti-criteria -- things that must not happen? That's how you catch regressions.
- And every single criterion has a verification method. Not "looks good" -- an actual test.
- This is how you prevent vague requirements from turning into vague output.

**Image**: SCREENSHOT FROM docs/workbench-presentation.html SLIDE 2 -- the pipeline Mermaid diagram showing 13 skills color-coded across OBSERVE (blue), THINK (purple), PLAN (orange), BUILD (green), VERIFY (red) with arrows showing the chain flow and shortcut paths.

**Speaker notes**:
- Here's the full pipeline. 13 skills across five phases.
- You start with research -- gather context on whatever you're building.
- Then you think: break it down to first principles, stress-test it with a red team, review the architecture.
- Then you plan: generate a requirements document with testable acceptance criteria.
- Then you build: implement against those criteria with a verify loop on each one.
- Then you verify: security-focused code review, then a final quality gate audit.
- There are shortcuts too. If you know the domain well, you can skip from research straight to the PRD.

### Slide 11: Example -- Compliance Report Generator

**Bullets**:
- /research: what exists, what's hard, what's solved
- /first-principles: irreducible requirements for compliance
- /red-team: failure modes, audit gaps, edge cases
- /create-prd: ISC-gated requirements document
- /implement-prd > /review-code > /quality-gate

**Image**: A vertical timeline/flowchart showing 7 steps, each with a skill name and a one-line output. Step 1: "/research -- Technical brief on compliance automation." Step 2: "/first-principles -- 5 core assumptions identified." Step 3: "/red-team -- 8 failure modes, 3 critical." Step 4: "/create-prd -- PRD with 6 ISC criteria." Step 5: "/implement-prd -- All 6 criteria verified." Step 6: "/review-code -- 0 critical findings." Step 7: "/quality-gate -- PASS."

**Speaker notes**:
- Let me walk through a concrete example. Say you need to build a compliance report generator.
- Step one: research. The system surveys what exists, what's hard, what tools are available.
- Step two: first principles. What are the irreducible requirements? What can't be negotiated?
- Step three: red team. What are the failure modes? What happens if the data source is stale? If the format changes?
- Step four: create a PRD. Six testable acceptance criteria, each with a verification method.
- Step five: build it. Each criterion gets implemented and verified individually.
- Step six: security-focused code review. Step seven: final quality gate audit.
- Every step produces an artifact. Every artifact is traceable. That's the point.
