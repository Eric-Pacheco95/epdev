## Producer Lifecycle

- "Idle Is Success" applies to producer OUTPUT (zero proposals/signals when thresholds aren't met = healthy). Silent producer detection applies to producer EXECUTION (zero runs for 2+ consecutive days with no error = suspect) — send Slack alert to `#jarvis-decisions` and suspend until Eric reviews
- Heartbeat auto-signals must require non-zero delta and meet min_delta thresholds — cumulative counters (failure_count, security_event_count) need delta >= 3 to avoid noise from single-count increments; use `min_delta` field in heartbeat_config.json
- Every verification/audit layer must emit its own health signal — if the verifier itself fails to execute, it must produce a louder alert than a verification failure; silent verifier failures create false confidence
- Synthesis threshold is set to 35 (hard ceiling) with tiers at 15/48h and 10/72h — lower ceiling to 15 when velocity drops below 3/day
- After any autonomous /absorb run (Slack poller Tier 1), verify the output chain: signal file exists, TELOS update is appropriate, audit trail is complete
- **Autonomous prompt requirements must pair the requirement with the named failure mode it prevents** — `"VERIFIER WILL FAIL with exit_reason=X if Y"` or `"WATCHDOG WILL KILL if Z"`. Advisory phrasing alone (`must`, `should`, `required`) is soft-overridden by the agent's own SKILL.md template. Test compliance on first run; if an agent skips the requirement, strengthen the language rather than re-running. Why: `2026-04-26_citation-override-insufficient-agent-ignored-weak-language.md` — three consecutive autonomous research runs ignored a "must include 3 citations" override until rephrased with named verifier consequence; agents weight named consequences much higher than property assertions.
- **Autonomous producers must dedup against non-terminal status before backlog injection.** Query for existing rows with the same `routine_id` OR description fingerprint in any active status (pending, pending_review, executing, verifying, claimed) — not just `done`. Dedup-against-done-only allows duplicate tasks to accumulate hourly on persistent failures. How to apply: add an open-row guard in `propose_task()` and `backlog_append()` callers; autoresearch pre-apply workflows must also verify the underlying condition is still true before generating proposals. Why: 8 identical auto-heal tasks injected hourly for the same already-fixed bug (2026-04-17); ISC producer re-proposed the same 5 criteria on consecutive days.
- **`producers.json` `state_file` paths and the producer script's actual output path must stay synchronized; treat producers.json as a secondary config that drifts.** When a producer script upgrades its output path, update producers.json in the same commit — not the next session. How to apply: any commit touching a producer's output path must also touch producers.json; consider a pre-commit grep gate that flags scripts in `producers.json` whose declared `state_file` path does not exist on disk after the commit. Why: heartbeat upgraded its output path (Phase 3E) without updating producers.json; watchdog reported stale data while the producer ran fine for days.
- **Weekly-cadence (or any > daily) producers require a per-producer `max_age_hours` override; the global 26h freshness threshold fires false-WARN every cycle on weekly tasks.** How to apply: any producer with cadence > daily must declare its own `max_age_hours` in producers.json (e.g., `168 + 6h slack` for weekly, `720` for monthly); the watchdog must read per-producer override before falling back to the global default. Why: 2026-04-25 producer health false-alarms on weekly tasks generated phantom incidents that obscured real failures.
- **The vitals/producer collectors are ground truth for infra state — operator memory and routing-table summaries drift from the actual count.** How to apply: before stating any infra count to Eric ("we have N dimensions / N producers / N skills") or acting on one in a PRD, run the collector or grep the registry — never quote from memory or routing tables. The router is a map, not the territory. Why: 2026-04-25 — operator believed 15 dimensions; actual count from collector was 6.

## Signal & Alert Design

- **Any alert cap must be paired with a re-alert interval config param** (e.g. `REALERT_INTERVAL_S = 86400`) that fires while the incident remains active — silence after cap is indistinguishable from resolution to the operator. Delivery counters must only increment on transport ACK, not optimistically before send; wrap all delivery calls in try/except that logs to a delivery-failure file and does NOT increment the counter on exception. Why: 2026-04-10 crypto-bot DMS fired 3 alerts then went silent for 205h with `alerts_sent=3` while actual deliveries were unverified — cap-without-cadence + counter-before-ACK compounded to make a 9-day incident invisible.
- **Autonomous runners assembling LLM prompts must enforce a hard 90K-char truncation gate before the `claude -p` call.** If the assembled prompt exceeds 90K chars, drop oldest signals first (end of the newest-first list), log dropped filenames, then proceed with the trimmed prompt — never warn-and-proceed on an oversized prompt. Why: two consecutive days of autoresearch produced zero output (Bun crash then 300s timeout) while the runner warned at 100K+ and continued anyway; advisory warnings with no structural gate are dead code.
- **Design alarms from failure mode first, not volume threshold.** Binary state resources (API status, stream health, plan type) require state-check alarms — not count/percentage thresholds. Threshold metrics are noise when the failure mode is qualitative (stream returns error, not "stream returns N fewer events"). How to apply: before adding a numeric threshold to any collector, ask "what does threshold=CRIT actually tell me?" — if the answer is "the resource is broken," use a state-check instead.
- Prediction signals (backtest, resolution, calibration) use their own synthesis cycle, not /synthesize-signals. Bulk batches (20+) must carry a domain category tag. `suspect_leakage: true` signals require Eric's review before contributing to calibration at full weight.
- **Before closing any producer incident, require a corroborating artifact (dispatcher JSON, dimension log entry, or outcome file) when incident streak and state disagree.** Streak-alone closure silently re-opens identical tasks on the next run. Why: producer health false-alarms on weekly tasks created phantom incidents that re-triggered hourly without artifact evidence.
- An autonomous producer is not "live" until it has produced outcome artifacts, not just run successfully — track what the producer creates (knowledge articles, scored predictions, merged branches), not whether the script exited 0; before updating TELOS or tasklist status, verify at least 1 outcome artifact exists in the last 7 days
- Alerting collectors that report shared-host metrics (TCP connections, memory, file handles) must attribute to specific processes by name+cmd, never blanket-blame a class like "Claude". How to apply: any new collector emitting alert text must call a top-N-holder helper (`tools/scripts/lib/net_util.py` for TCP); reject generic "close X sessions" templates in favor of named-process attribution
- **Any absorbed item with `status: APPROVED` that is ≥7 days old without a corresponding TELOS update must generate an `external_monitoring`-dimension task in the overnight runner.** `status: APPROVED` means the content was committed for TELOS integration — it is not optional. Why: `2026-03-30_geo-strategy-iran-trap.md` sat with 3 APPROVED proposals for 27 days; `2026-04-02_ai-bubble-datacenter-delays.md` sat with 2 PENDING proposals for 24 days; both were only discovered during a manual synthesis pass. The absorbed→TELOS pipeline produces but does not flush without a downstream trigger.
